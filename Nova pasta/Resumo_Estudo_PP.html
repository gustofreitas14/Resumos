1. Conceitos Fundamentais

Processamento Paralelo: forma de computação onde os cálculos são realizados ao mesmo tempo, dividindo uma tarefa grande em partes menores que podem ser resolvidas simultaneamente em múltiplos núcleos/processadores.

Definições rápidas

Sequencial / Série

Um único fluxo de execução. Uma instrução termina antes da próxima começar. Simples, mas não aproveita múltiplos núcleos.

Concorrência

Várias tarefas progridem no mesmo intervalo de tempo, intercaladas. Pode ocorrer em 1 só núcleo. Foco: gerir múltiplas tarefas.

Paralelismo

Execução realmente simultânea em múltiplos núcleos/processadores. Requer hardware multi-core. Foco: reduzir tempo total.

Assíncrono

Uma tarefa não bloqueia a outra enquanto espera por I/O. Não significa necessariamente paralelo.

Distribuído

Tarefas executadas em múltiplas máquinas ligadas por rede; cada uma com a sua memória.

SMP vs AMP

SMP (Simétrica): processadores equivalentes, partilham memória.
AMP (Assimétrica): 1 processador coordena, outros executam papéis específicos.
Regra de ouro: Toda execução paralela é concorrente, mas nem toda execução concorrente é paralela.
CaracterísticaConcorrênciaParalelismo
ExecuçãoIntercaladaSimultânea
Hardware1 núcleo bastaMúltiplos núcleos
ObjetivoGerir tarefas, melhorar responsividadeAcelerar execução, aumentar throughput
ExemploServidor web atende vários clientesProcessar vídeo em múltiplos núcleos

2. Processo vs Thread

PROCESSO

Instância independente de um programa em execução, com espaço de memória próprio.
  • Não partilha memória por defeito
  • Comunicação via IPC (Queue, Pipe, sockets, ficheiros)
  • Criação/encerramento mais pesado
  • Isolamento total entre processos
  • Contorna o GIL → paralelismo real
VS

THREAD

Menor unidade de execução dentro de um processo. Múltiplas threads partilham memória.
  • Partilha código, dados e ficheiros do processo
  • Comunicação rápida via memória partilhada
  • Leve e rápida de criar
  • Requer sincronização (locks)
  • Limitada pelo GIL em CPython
Analogia: Um processo é como uma empresa com edifício próprio e recursos isolados. As threads são os empregados dessa empresa que partilham o mesmo escritório, ficheiros e recursos, mas trabalham em tarefas diferentes.
CaracterísticaMultithreadingMultiprocessing
MemóriaPartilhadaSeparada (isolada)
GILLimitado pelo GILNão afetado (interpretador próprio)
OverheadBaixoAlto (criação de processos)
ComunicaçãoSimples (memória partilhada)Complexa (Queue, Pipe)
Caso idealI/O-bound rede, disco, BDCPU-bound cálculos pesados, ML
AgendamentoInterpretador PythonSistema Operativo
RiscosRace conditions, deadlocksOverhead, IPC complexo

3. Global Interpreter Lock (GIL)

O que é: O GIL é um mutex no CPython que protege o acesso a objetos Python, permitindo que apenas uma thread execute bytecode Python de cada vez, mesmo em sistemas multi-core.

Threading + CPU-Bound

Não há paralelismo real. O GIL bloqueia outras threads. Pode até ser mais lento que sequencial devido ao overhead.

Threading + I/O-Bound

Funciona bem. O GIL é libertado enquanto a thread espera por I/O, permitindo que outras threads executem.

Multiprocessing

Solução para CPU-bound. Cada processo tem o seu próprio interpretador e o seu próprio GIL → paralelismo real.
Python 3.13+: Introdução de build experimental "free-threading" (no-GIL) para permitir paralelismo completo com threads no futuro.
Regra prática:
• I/O-bound (rede, disco, espera) → multithreading ou asyncio
• CPU-bound (cálculo pesado em Python puro) → multiprocessing

5. Modelos de Memória

Memória Partilhada

Todos os processadores partilham um único espaço de endereçamento global.
  • Comunicação rápida e implícita (leitura/escrita)
  • Programação mais simples
  • Requer sincronização (locks) → evitar race conditions
  • Menos escalável (limite físico)
  • Tipos: UMA (Uniform Access), NUMA (Non-Uniform Access)
VS

Memória Distribuída

Cada processador tem o seu próprio espaço de memória privado.
  • Comunicação explícita via troca de mensagens (rede)
  • Altamente escalável (base de clusters/supercomputadores)
  • Sem contenção de memória
  • Programação complexa, overhead de comunicação
  • Sem acesso direto à memória de outros nós

6. Tipos de Paralelismo

Paralelismo de Dados

Divide os dados em partes; a mesma operação é aplicada a cada parte simultaneamente.
Ex: somar uma matriz grande dividida em blocos; aplicar filtro a uma imagem por tiles.
Requer balanceamento de carga.

Paralelismo de Tarefas

Distribui diferentes tarefas/funções por processadores. Tarefas podem ser independentes ou requerer sincronização.
Ex: motor de jogo — thread A física, thread B IA, thread C renderização.

Paralelismo de Instruções

Várias instruções executadas ao mesmo tempo dentro do processador.
Ex: pipeline, execução fora de ordem.

Paralelismo de Bit

Vários bits de dados operados ao mesmo tempo.
Ex: CPU de 64 bits manipula 64 bits em uma única operação.

Padrão Master/Worker

Um processo Master divide a tarefa global em subtarefas e distribui pelos processos Worker.
  1. Inicialização — Master prepara o ambiente e os workers
  2. Distribuição — Master divide e envia tarefas aos workers
  3. Processamento — Workers executam em paralelo, independentemente
  4. Agregação — Master recolhe e combina os resultados parciais
Vantagens: simplicidade, gestão centralizada, balanceamento dinâmico.
Desvantagens: Master pode tornar-se um gargalo (bottleneck).

Exemplo prático — 3 pedidos de 1 segundo cada

Síncrono 
for pedido in [1, 2, 3]:
    consultar_pedido(pedido)
# Tempo total: ≈ 3.0 s
→
Assíncrono (asyncio) 
tarefas = [consultar(p) for p in [1,2,3]]
await asyncio.gather(*tarefas)
# Tempo total: ≈ 1.0 s
Pergunta-chave para escolher: O tempo está a ser gasto a calcular (síncrono/multiprocessing) ou a esperar (assíncrono/threading)?

multiprocessing — Pool.map

import multiprocessing
import time

def calcular_quadrado(n):
    time.sleep(1)
    return n * n

if __name__ == "__main__":
    numeros = [1, 2, 3, 4, 5, 6, 7, 8]
    with multiprocessing.Pool(processes=4) as pool:
        resultados = pool.map(calcular_quadrado, numeros)
    print(resultados)
# Sequencial: 8s | Paralelo (4 cores): ~2s

threading — sincronização com Lock

import threading

contador = 0
lock = threading.Lock()

def incrementar():
    global contador
    for _ in range(100000):
        with lock:               # Secção Crítica
            contador += 1

threads = [threading.Thread(target=incrementar) for _ in range(5)]
for t in threads: t.start()
for t in threads: t.join()
print(f"Contador: {contador}")

asyncio — async/await + gather

import asyncio, aiohttp

async def fetch(session, url):
    async with session.get(url) as r:
        return await r.text()

async def main():
    urls = ['http://example.com', 'http://python.org']
    async with aiohttp.ClientSession() as s:
        tasks = [fetch(s, u) for u in urls]
        respostas = await asyncio.gather(*tasks)

asyncio.run(main())

concurrent.futures — ThreadPoolExecutor

from concurrent.futures import ThreadPoolExecutor

with ThreadPoolExecutor(max_workers=2) as executor:
    future1 = executor.submit(tarefa, 2)
    future2 = executor.submit(tarefa, 1)
    print(future1.result())   # bloqueia até estar pronto
    print(future2.result())
asyncio essenciais:
  • async def — define uma corrotina
  • await — suspende a execução sem bloquear
  • asyncio.gather(*tasks) — executa corrotinas em concorrência
  • asyncio.run(main()) — inicia o event loop
  • Event Loop — núcleo da execução; distribui tarefas e callbacks
  • Em asyncio NÃO usar time.sleep — usar await asyncio.sleep()

9. Mecanismos de Sincronização

Lock (Mutex)

Garante exclusão mútua. Apenas uma thread possui o lock de cada vez. Protege secções críticas.

Semaphore

Mantém um contador interno. Permite que N threads acedam a um recurso simultaneamente (ex: limitar conexões BD).

Event

Comunicação simples: uma thread sinaliza (set), outras esperam (wait) até o evento ocorrer.

Condition

Mais avançado que Event. Threads esperam por uma condição específica e são notificadas (notify/notify_all).

Barrier

Ponto de sincronização em grupo. Bloqueia N threads até todas chegarem; depois libertam-se simultaneamente.
Boa prática: usar sempre context manager (with lock:) para garantir libertação do lock mesmo em caso de erro → evita deadlocks.

10. Métricas de Desempenho

Speedup (Aceleração)

S = Tsequencial / Tparalelo
Mede o ganho de velocidade ao paralelizar

Linear (S ≈ P)

Cenário ideal. Duplicar processadores → duplicar velocidade.
Causa: tarefas embarrassingly parallel sem comunicação.

Sub-linear (S < P)

Cenário mais comum. Ganho inferior ao aumento de recursos.
Causa: overhead de comunicação, sincronização, parte sequencial.

Super-linear (S > P)

Anomalia rara. Ganho superior ao número de processadores.
Causa: efeitos de cache (dataset cabe na L1/L2 de cada processador).

Eficiência

E = S / P
S = Speedup, P = Nº de processadores. Métrica entre 0 e 1.
FaixaAvaliaçãoSignificado
E ≈ 1.0 (100%)Ideal / LinearCada processador adicionado contribui 100%. Raro.
0.7 < E < 0.9Alta EficiênciaExcelente utilização, overhead baixo.
E < 0.5Baixa EficiênciaSaturação. Adicionar mais processadores não ajuda.
Trade-off fundamental: à medida que P aumenta, Speedup aumenta mas Eficiência diminui (overhead de gestão).

11. Lei de Amdahl ⭐

Speedup = 1 / ((1 - P) + P/N)
  • P — Fração do código que é paralelizável (0 a 1)
  • N — Número de processadores
  • (1 − P) — Fração sequencial (o gargalo)
O Limite Teórico: Mesmo com processadores infinitos (N → ∞), o speedup máximo é limitado pelo inverso da parte sequencial:
Smax = 1 / (1 − P)

O "Muro" do Desempenho — Tabela de referência

Fração Paralela (P)Fração Sequencial (1-P)Speedup Máximo
50%50%2×
75%25%4×
90%10%10×
95%5%20×
99%1%100×
Interpretação: Se 5% do programa não puder ser paralelizado, nunca conseguirá acelerar o programa mais de 20×, independentemente do hardware.

14. Desafios Comuns em Programação Paralela

🔴 Deadlock (Impasse)

Duas ou mais threads bloqueadas indefinidamente, cada uma à espera de um recurso que a outra possui. O programa "congela".
Solução: definir ordem estrita para aquisição de locks; usar timeouts.

🟠 Race Condition

O resultado depende da ordem aleatória de execução das threads. Leva a dados corrompidos e bugs difíceis de reproduzir.
Solução: sincronização (Locks, Semáforos); operações atómicas.

🟣 Starvation

Uma thread nunca consegue acesso aos recursos porque outras threads "gulosas" (maior prioridade) os ocupam constantemente.
Solução: algoritmos justos (Fairness); Priority Aging.

🔵 Overhead Excessivo

O custo de criar/gerir/sincronizar threads supera o ganho da paralelização → programa mais lento.
Solução: aumentar granularidade das tarefas; usar Thread Pools.

Quando o processamento em série pode ser MAIS rápido?

  • Overhead de comunicação/sincronização excede o ganho
  • Problemas pequenos — o custo de dividir não compensa
  • Dependências fortes entre passos — não há o que paralelizar
  • Hardware limitado — poucos núcleos disponíveis
  • Cache locality — serial faz uso mais eficiente do cache

Boas Práticas

Profiling Primeiro

Nunca otimize prematuramente. Identifique gargalos reais antes de paralelizar.

Abstração Elevada

Comece com joblib / concurrent.futures antes de gerir manualmente threads/processos.

Minimizar Estado

Evite variáveis globais e estado partilhado. Prefira troca de mensagens / dados imutáveis.

Granularidade

Cuidado com overhead. Criar threads para tarefas minúsculas pode ser mais lento. Agrupe tarefas (chunking).

Gestão de Locks

Use sempre context managers (with lock:) para garantir libertação.

Conhecer Hardware

Distinguir CPU-bound (multiprocessing) de I/O-bound (threading/asyncio).
Regra de Ouro (Go Proverb): "Não comunique partilhando memória; partilhe memória comunicando."

15. Exemplos & Padrões em Sistemas Distribuídos (Sockets)

Padrões de Servidor TCP

1. Servidor Iterativo (baseline)

Trata um cliente de cada vez. Bom para protótipos; degrada com muitos clientes. 
while True:
    conn, addr = s.accept()
    handle_client(conn)

2. Thread Pool (I/O concorrente)

Submete cada ligação a um ThreadPoolExecutor. Ideal para I/O-bound. 
with ThreadPoolExecutor(max_workers=W) as pool:
    while True:
        conn, _ = s.accept()
        pool.submit(handle_client, conn)

3. Híbrido (threads + processos)

Threads tratam I/O; processos tratam cálculo pesado. Recomendado para I/O + CPU. 
with ProcessPoolExecutor() as proc_pool:
    with ThreadPoolExecutor() as io_pool:
        while True:
            conn, _ = s.accept()
            io_pool.submit(handle, conn, proc_pool)

4. Thread-per-connection

Uma thread por ligação. Simples mas escala mal com milhares de clientes.

Padrão Map-Reduce (distribuído)

map(nó, bloco):
    para cada palavra p:
        emitir (p, 1)

reduce(p, contagens):
    devolver soma(contagens)
